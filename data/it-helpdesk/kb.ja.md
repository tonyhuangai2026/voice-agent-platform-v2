# ITヘルプデスク・ナレッジベース · Amazon Connect

> ITSCチーム向けの音声優先ソリューション。
> Amazon Connect + Nova Sonic(エンドツーエンド音声モデル)+ AWS Lambda + Case (ITSM)。
> 本人確認はSMS OTP — 声紋登録は不要。

---

## 1 · ソリューション概要

設計原則:**音声優先 → 意図検出 → 可能ならセルフサービス → 不可なら丁寧にエスカレーション**、
全工程で自動チケット化。

### 1.1 3つの柱

- **音声入口と本人確認** — Amazon ConnectがPSTN/SIP着信を終端。
  本人確認は **SMS OTP**(社員番号 + 登録携帯 + 6桁ワンタイムコード)。
  シンプルで信頼性が高く、ユーザー側の事前準備ゼロ。

- **Nova Sonic エンドツーエンドモデル** — Amazon Nova Sonic は
  ASR + NLU + TTS を一体化した音声モデル。自然な韻律、低レイテンシー、
  バージイン対応。従来の Lex + Polly 二段構成を置き換え。

- **自動化とチケット** — AWS Lambda が Active Directory / Entra ID で
  ロック解除を実行し、Case (ITSM) でインシデントを自動作成・更新・解決。

### 1.2 主要KPI

| 指標 | 目標 |
|---|---|
| セルフサービス率 | ≥ 65% |
| 平均通話時間 | < 90秒 |
| チケット自動化率 | 100% |
| P1 引取時間 | < 30秒 |

## 2 · リファレンスアーキテクチャ

### 2.1 エンドツーエンド呼フロー

```
社員 → PSTN/SIP → Amazon Connect (Contact Flow / IVR)
                        │
                        ├─► Nova Sonic (E2E 音声 LLM, ASR+NLU+TTS)
                        │       │
                        │       └─► AWS Lambda (業務オーケストレーション)
                        │              ├─► Active Directory / Entra ID
                        │              ├─► Case (ITSM)
                        │              └─► DynamoDB (OTP / セッション)
                        │
                        ├─► Pinpoint / SNS (SMS OTP 配信)
                        ├─► Kinesis Video Streams ─► S3 (録音アーカイブ)
                        └─► CloudWatch + Contact Lens (分析)

       │ ルーティング判定
       ├─► セルフ完結 → 自動クローズ(終話)
       └─► 有人必要   → Agent Workspace (CCP + スクリーンポップ)
```

### 2.2 サービススタック

| サービス | 役割 |
|---|---|
| Amazon Connect | IVR · ルーティング · オペレーターデスクトップ |
| Nova Sonic | E2E 音声モデル |
| AWS Lambda | AD ロック解除 · チケット · オーケストレーション |
| Pinpoint / SNS | SMS OTP 配信 |
| Contact Lens | 通話分析 · 感情分析 |
| Case | ITSM チケットシステム |
| DynamoDB | OTP保管 · セッション |
| S3 + KMS | 録音アーカイブ · 保管期限 |

## 3 · 5つのデモシナリオ

ITSC で頻度の高い5つの呼パターン。それぞれ呼フロー、自動化、
エスカレーション戦略が異なります。

### 3.1 シナリオ1 · 電話セルフサービスでロック解除(完全自動)

**トリガー**:発信者がアカウントロック解除を希望、かつ電話セルフサービスの
要件を満たす(登録携帯あり、ポリシー許可)。
**ルートタグ**:● 完全自動 · S2S モデル。

#### フロー

```
1. 着信 → Connect contact flow → Nova Sonic 双方向 S2S セッション開始。
2. Nova Sonic:「お電話ありがとうございます、ITサービスデスクです。」
3. お客様:「アカウントがロックされてしまったので、解除してほしい。」
4. Nova Sonic 意図 = UnlockAccount、empId スロット収集。
5. 社員番号を伺い、1文字ずつ復唱して確認。
6. Lambda fn-send-otp:
     - AD から登録携帯を取得
     - Pinpoint/SNS で 6桁 OTP を送信
     - DynamoDB に保存(TTL 5分、最大3回)
7. Nova Sonic がお客様に SMS の 6桁コードを読み上げてもらう。
8. Lambda fn-ad-unlock が Graph API / PowerShell で AD ロック解除。
9. 同じ Lambda が Case REST API でインシデントを作成・自動解決、
    Category = Access/Unlock、Caller = 認証済み社員。
10. Nova Sonic がチケット番号を復唱;SNS で確認 SMS;通話終了。
```

#### キーパラメータ

- 本人確認:SMS OTP
- OTP有効:5分
- 最大試行:3回
- 音声エンジン:Nova Sonic
- ゴール:有人ゼロ、< 90秒、声紋登録不要

#### 対話例

> Sam:「お電話ありがとうございます、ITサービスデスクのSamと申します。
>      本日はどのようなご用件でしょうか?」
> お客様:「アカウントがロックされてしまって。」
> Sam:「承知いたしました、解除いたします。社員番号をお願いします。」
> お客様:「W123456です。」
> Sam:「ご確認ですが、W・1・2・3・4・5・6でお間違いないでしょうか?」
> お客様:「はい。」
> Sam:「ただいまお電話に6桁のSMSコードをお送りしました。
>      読み上げていただけますでしょうか?」
> お客様:「8 2 4 1 5 7。」
> Sam:「ありがとうございます…解除完了いたしました。チケット番号
>      INC・0・0・1・2・3・4・5でございます。他にお困りの点は?」

### 3.2 シナリオ2 · バーチャルエージェントへ案内

**トリガー**:ロック解除の要望だが、電話セルフサービスの**要件を満たさない**
(登録携帯なし、ポリシーで強MFA要、ハイリスクユーザーグループ)。
**ルートタグ**:● デジタルチャネルへ。

#### フロー

```
1. 着信 → Nova Sonic 意図 = UnlockAccount。
2. Contact Flow が contact attributes をチェック → 電話セルフ不可。
3. Nova Sonic:「Teams または SMS にセキュアなリンクをお送りします。
   そちらで解除を完了させていただけます。」
4. Lambda が SNS SMS + Teams ボットでワンタイムディープリンクを送信
   (TTL 5分)。
5. Virtual Agent (Copilot Studio / Amazon Q Business) の「解除」
   トピックが引き継ぎ、MFA チャレンジ → 解除続行。
6. VA は**同じ** fn-ad-unlock Lambda を呼ぶ — 電話とデジタルで
   同じ動作・同じチケット分類体系。
7. チケット番号と監査証跡は元の Connect contact に attributes として
   書き戻し、統一分析可能。
```

#### なぜ電話セルフを強要しないか

音声 + AD ロック解除は強い本人確認が必要。シンプルなロック解除は
SMS OTP で十分ですが、ハイリスクなユーザーにはデジタルチャネルで
より強い MFA チャレンジを実施します。

### 3.3 シナリオ3 · MFA 再登録のお問い合わせ — 教育 + デジタルガイド

**トリガー**:お客様が MFA の再登録 / 新しいデバイスへの切替方法を質問。
**ルートタグ**:● 教育 + デジタル配信。

#### なぜ電話で完結しないか

新規 MFA デバイスのバインドは**機微な操作**で、純粋な音声チャネルでは
リスクが大きい。代わりに:口頭で要点 + 詳細手順を Teams + SMS に送付。

#### フロー

```
1. お客様:「スマホを変えたので MFA を再設定したい。」
2. Nova Sonic 意図 = MfaReRegister。
3. Sam が要点3つを音声でお伝え:
     - アカウントポータルで旧デバイスを削除
     - aka.ms/mfasetup を開く
     - 新しい認証アプリで QR コードをスキャン
4. Sam:「詳細な手順をお送りしましょうか?」
5. ご了承後、Lambda が Teams adaptive card + KB ディープリンクを送信。
6. Case で Service Request(Incident ではなく)を起票、Awaiting user。
7. Sam:「Teams にお送りしました。チケット番号 SR・0・0・0・9・8・7
   でございます。何かありましたら再度お電話ください。」
8. 48時間で完了しない場合、自動でオペレーターのコールバック起動。
```

### 3.4 シナリオ4 · お客様が有人をご希望 — 丁寧なエスカレーション

**トリガー**:
- 「人と話したい」「担当者に代わって」「オペレーター」、または
- Nova Sonic で連続2回 no-match、または
- 感情が2ターン以上ネガティブ継続。
**ルートタグ**:● 有人エスカレーション。

**原則**:セルフサービスを強要しない。「いつでも『担当者に代わって』
とおっしゃってください」というフォールバックは全通話を通じて有効。

#### フロー

```
1. Nova Sonic が SpeakToHuman 意図を検出(またはフォールバック発動)。
2. Sam:「かしこまりました、担当者にお繋ぎいたします。」
3. Contact flow が contact attributes に書き込み:
     empId, lastIntent, otpStatus, ticketId, transcriptUri
4. スキル + 優先度でキューにルート:
     - Account / Network / Endpoint
     - VIP / 通常
5. オペレーター CCP が応答;CTI アダプターのスクリーンポップで
   Case のチケットが事前入力 — 第一声から文脈を把握できる。
6. 通話後処理(ACW)で自動補完、オペレーターは必要箇所のみ修正。
```

#### トーン

落ち着いた、お詫びを含む、押し付けない口調。「ボットでは解決できず
申し訳ございません、すぐに担当者にお繋ぎいたします。」

### 3.5 シナリオ5 · 重大インシデント(P1) — エクスプレスレーン

**トリガー**:お客様が緊急キーワードを使用:
- 「本番が落ちている」「production down」「production outage」
- 「全社で使えない」「オフィス全体が影響」
- 「critical」「emergency」「緊急」
- Contact Lens 二次シグナル:感情 = 強いネガティブ。

**ルートタグ**:● P1 エクスプレスレーン。

#### フロー(セルフサービスを完全スキップ)

```
1. お客様:「本番環境が完全に止まっています!」
2. Nova Sonic 意図 = ReportIncident, severity = CRITICAL。
3. Contact Lens で確認:感情 強いネガティブ。
4. **すべてのセルフサービスをスキップ** — 社員番号や非必須情報は
   問わない、速度優先。
5. Lambda 並列 fan-out:
     - Case:P1 Incident 起票、War Room 自動作成
     - PagerDuty:オンコール起動(P1)
     - Teams:戦時室チャネル
     - エグゼクティブ通知
6. Sam:「P1 にエスカレーションしました。チケット番号は
   INC・0・0・9・9・0・0・1、ただいま当番マネージャーへお繋ぎ
   いたします。」
7. Major Incident Manager スキルグループへ直接接続、通常キュー
   スキップ、SLA < 30秒。
8. Contact Lens がリアルタイム転写を MIM にストリーム — お客様は
   同じ説明を繰り返す必要なし。
9. 事後:録音 + 転写を S3 にアーカイブ、PIR(事後レビュー)に紐付け。
```

## 5 · セキュリティ、コンプライアンス、可観測性

### 5.1 SMS OTP による本人確認

- 社員番号 + 登録携帯 + 6桁 OTP
- TTL 5分、最大3回
- レート制限:1社員あたり10分1通
- 声紋登録不要、ユーザー導入コストゼロ

### 5.2 録音と転写

- S3 サーバーサイド暗号化 + KMS CMK
- Object Lock 保管期限(規制対応)
- Contact Lens で PII(カード番号、PIN、マイナンバー等)を自動マスキング

### 5.3 可観測性

- CloudWatch ダッシュボード:セルフ率、放棄率、キュー待機、P1引取時間
- QuickSight:ITSC マネージャーへ週次レポート
- 通話単位ドリルダウン:Connect contact ID → Lambda ログ → Case →
  Contact Lens 転写
